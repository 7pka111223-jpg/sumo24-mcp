# Offline HTML Tool — House Pattern

Every tool in this workflow is a browser app that must work with **zero
network access, zero install, zero backend**. These rules are the difference
between a tool that works on a locked-down site laptop / an iPhone home
screen and one that silently breaks.

## Hard rules (non-negotiable)

1. **No network at runtime. Ever.** No CDN `<script src>`, no Google Fonts,
   no fetch to external hosts. Every library, font, icon, and ML model ships
   in the repo (`vendor/` directory) or is inlined. Before finishing, grep
   the output for `https://` and `http://` and justify every hit.
2. **Two distribution shapes, kept in sync.** Most tools ship BOTH a
   directory app (`index.html` + assets, servable and PWA-installable) and a
   single self-contained file (everything inlined — copy anywhere,
   double-click). When you change one, rebuild the other in the same commit.
   The single file is generated by a build script (`node tools/build-*.js` /
   `npm run build`), never hand-edited.
3. **`file://` must work.** Some browsers restrict workers/modules from
   `file://`. Structure code so double-click works in Chrome/Edge; where a
   worker is unavoidable, provide `serve.sh` / `serve.bat`
   (`python3 -m http.server`) as a documented fallback — never require it.
4. **PWAs: bump the service-worker cache version on every asset change.**
   Change `index.html` without bumping the `CACHE_NAME` in `sw.js` and every
   installed iPhone keeps serving the stale version indefinitely. This is
   the #1 recurring bug in this workflow.
5. **All processing is client-side.** Files the user opens are read locally
   (FileReader / drag-drop). State no-upload explicitly in the UI — these
   tools handle sensitive engineering and banking documents.

## Standard structure

```
index.html                 # app entry (directory shape)
app.js / src/              # logic; ES modules only if a build step exists
styles.css
vendor/                    # embedded third-party libs + models, committed
tools/build-single-file.js # inlines everything into <name>.standalone.html
<name>.standalone.html     # generated single-file twin, committed
sw.js + manifest.json      # only if installable PWA
```

## Platform specifics

- **Target: mobile Safari + desktop Chrome/Edge.** Test touch targets,
  safe-area insets (`env(safe-area-inset-*)`), and Add-to-Home-Screen flow
  for PWAs.
- **File saving:** prefer the File System Access API (Chrome/Edge, folder
  picker) with a plain `<a download>` fallback everywhere else. Never assume
  the picker exists.
- **Big binary assets** (OCR models, dictionaries): store under
  `vendor/embed/` and lazy-load from the local path, not inline in the HTML
  of the directory shape (the standalone build may inline them).
- **Fonts and icons:** embed as data URIs or local files; a missing external
  font silently changes layout offline.

## Verification checklist before finishing

- [ ] DevTools Network tab shows zero external requests after load
- [ ] Works opened via `file://` double-click (or the serve fallback is
      documented in README)
- [ ] Standalone twin rebuilt and byte-diff sanity-checked
- [ ] PWA: `sw.js` cache version bumped; verified update flow
- [ ] Existing tests pass (`npm test` where present)
